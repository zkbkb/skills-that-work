### 设计决策

SKILL.md 作为 Claude 的指令文档，需要同时承载两个职能：(1) 告诉 Claude 怎么跟用户互动；(2) 告诉 Claude 怎么执行计算。问题在于 Claude 可能将 (2) 的内容泄露到 (1) 的场景中，导致用户看到脚本路径、引擎来源、模块名等幕后信息。

解决方案：不删除技术指令（Claude 执行时确实需要），而是在文档层面建立明确的信息边界——最高优先级的边界规则 + 逐节的「内部执行」标注 + 面向用户文本的措辞泛化。三层防护确保 Claude 在对话中只呈现易学视角，不暴露工程实现。

### 设计决策记录

本次合并源于一个观察：v1.0.1 的示例驱动方法论在精度上表现好（Claude 可以直接索引到答案），但覆盖面受限于预设场景；v1.1.0 的规则驱动方法论泛化性强，但在语义敏感的场景下（如感情卦中"体克用"被过度乐观解读）Claude 会做出偏离易学共识的推断。根本原因是六亲动态吉凶框架的缺失。

解决方案：规则层回答"分析什么"，校准层回答"分析到什么精度"。校准层不作为独立章节，而是嵌入到对应的规则章节中（标记为 🔸），确保 Claude 在读到通用规则时紧接着就能看到场景化的精度校准。
